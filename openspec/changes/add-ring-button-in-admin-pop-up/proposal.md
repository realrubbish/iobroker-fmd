## Why

The adapter's whole purpose is "ring an FMD device on demand". Today the user can only trigger a ring via the hardware button (Shelly `triple_push`) or by writing `0_userdata.0.FindMyDevice.ring.<deviceId> = true` from another script — neither of which is reachable from the ioBroker admin wrench pop-up. The 2026-06-11 E2E test session confirmed that the deployed admin stack (admin 7.7.22 / js-controller 7.1.2) renders the **native** `jsonConfig.json5` form in the pop-up, not the Vite SPA, so the only realistic delivery surface for a new trigger is a `type: "sendTo"` form item backed by the existing `onMessage.ring` adapter handler.

The earlier change `add-or-fix-test-button-in-admin-pop-up` (commit 72dc1bf) shipped a `Test Connection` button using the same delivery surface. That button was the wrong action for a "Test Connection" label — the test is useful for debugging but the pop-up is the right place for the **ring trigger** (the most common operation), and the ring handler is already implemented end-to-end. This change **supersedes** `add-or-fix-test-button-in-admin-pop-up` by re-purposing the schema slot to `ring` and dropping the testConnection schema item. The old change folder is left on disk as a historical record (its tasks are all done — the display-fix in `App.tsx` is the only lasting code change from it).

## What Changes

- **Replace** the `testConnection: { type: "sendTo", command: "testConnection" }` schema item in `src-admin/schema.json5` with `ringNow: { type: "sendTo", command: "ring" }` (Label: `Ring Device`). The `command` is the same `ring` string the adapter's `onMessage.ring` handler already accepts.
- **Send the configured deviceId in the payload.** The ConfigSendTo widget sends `socket.sendTo(adapterName.instance, command, payload)`. We pass `payload: { deviceId: config.ringDeviceId }` so the ring handler knows which device to ring. The `onMessage.ring` handler in `src/main.ts` (lines 505-555) already accepts `obj.message?.deviceId` and dispatches the ring.
- **Remove the `lastTestResult` staticText placeholder** that paired with the testConnection button. It is no longer used (the new ring path returns its reply via `window.alert` and does not need a staticText line). The other Status panel items (`connectionState`, `lastError`) stay.
- **Keep the `App.tsx` `hasSchemaTestConnection` guard logic** but rename it to `hasSchemaRingNow` and re-target the schema probe. The guard is still relevant for the iframe path (a future admin version that mounts the Vite SPA), where the same double-render avoidance applies.
- **Keep the App.tsx custom `<button>` as a fallback** for the iframe path. It is renamed to "Ring Device" and the call is `socket.sendTo("iobroker-fmd.0", "ring", { deviceId: config.ringDeviceId })`. Gated on `!hasSchemaRingNow`.
- **Update docs** (`docs/admin-ui.md`, `docs/admin-ui-investigation-2026-06-08.md`, `docs/docker-development.md`) to reflect the new "Ring Device" button and the deprecation of the test-connection path.
- **Carry forward the `display`-property fix from the previous change** — `App.tsx` `connectionState` push no longer sets the `display` key (that property was rejected by the jsonConfig meta-schema and caused the whole Status panel to be skipped in the admin SPA render). The previous change's commit `72dc1bf` includes a stale `connectionState.display` block in `src-admin/App.tsx` (it was removed in the local working tree but never re-committed/pushed). The fix is re-asserted here.
