## Why

The Admin-UI's Connection Status panel currently surfaces `info.connection` and `info.lastError` (polled every 5 s), but the user has no way to **proactively** verify their credentials without restarting the adapter and reading container logs. The `admin-ui` capability already mandates a `Test Connection` button in the Status panel (`admin-ui/spec.md`, Requirement "Test Connection button is present") and `src/main.ts` already implements the `testConnection` message handler — only the form-side `sendTo` schema item is missing. This was explicitly deferred in `admin/jsonConfig.json5` ("add it in a follow-up change once we have the meta-schema for the result sub-object confirmed"). Closing that gap turns the wrench pop-up into a self-contained diagnostic tool and removes the last reason to tail `docker logs iobroker-fmd-dev` after saving credentials.

## What Changes

- Add a `Test Connection` button to the `Connection Status` panel of the Admin-UI form (driven by `JsonConfig`'s `type: "sendTo"` item), wired to the existing `testConnection` adapter message.
- Add a small "test result" line under the `Last Error` field that shows the most recent test outcome (`OK – connected` / `Failed – <reason>`) for at least one polling cycle, then reverts to the live `info.lastError` value.
- Update the "Status reflects an error" and "Devices panel" `admin-ui` requirements so the existing 5 s polling loop is documented to also clear the test result line on a fresh `info.lastError` change.
- Re-run `npm run build:admin` and commit the regenerated `admin/index.html` / `admin/index_m.html` / `admin/assets/` so the Docker dev container picks up the new form without a Node toolchain.
- Re-run `scripts/ring-smoke.mjs` and `scripts/auth-smoke.mjs` against `fmd.example.com` from the dev host to confirm the existing `testConnection` path still works end-to-end.

The underlying `testConnection` message handler does not change — this is a UI-side delta only.

## Impact

- **Code**: `src-admin/schema.json5` (add the `sendTo` item + the "last test result" static-text line); `src-admin/App.tsx` (push the latest test result into the `data` object on each 5 s poll).
- **Built artefacts**: `admin/index.html`, `admin/index_m.html`, `admin/assets/**` — regenerated by `npm run build:admin`, committed alongside the source change.
- **Backend**: `src/main.ts` `onMessage.testConnection` is already present and correct; no functional changes there. The handler's existing `{ success, message }` / `{ error }` reply shape is the contract the UI consumes.
- **Smoke tests**: `scripts/auth-smoke.mjs` and `scripts/ring-smoke.mjs` — re-run as the verification gate documented in `CLAUDE.md` step 0. No new scripts.
- **Schema**: no `io-package.json` changes; the button is purely a UI construct that calls the existing `testConnection` message.
